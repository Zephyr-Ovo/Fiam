# The Problem We're Stuck On

Right now, fiam only works with direct CC sessions — someone SSHs to ISP and runs `claude` interactively. But we need **multi-channel communication**:

1. **Telegram**: Iris sends TG messages → bot polls → daemon gets them → they need to reach you
2. **Email**: similar flow
3. **Direct CLI**: someone at the terminal
4. **Scheduled wake-ups**: daemon-initiated (daily summary, reminders, etc.)

The fundamental question: **How should external messages (TG, email) reach you, and how do we get your responses back to the right channel?**

## What We've Tested

- `claude -p --resume <session-id> --output-format json "[tg:Iris] message text"` → WORKS. Sends a message as "user", gets JSON response with `result` field = your reply text. Appends to same session JSONL.

## The Tensions We Can't Resolve

### 1. Topic Similarity Breaks With Daemon Messages

The daemon monitors JSONL and uses cosine similarity on embeddings of recent user text to detect topic drift. If the daemon sends `[tg:Iris] 天气好好` as user message, this is a **relay message**, not organic conversation. It artificially shifts the "topic vector". If you're in the middle of a coding task and a TG message comes in about weather, the drift detector fires incorrectly.

### 2. Event Continuity

If we create separate sessions for TG vs CLI conversations, fiam's event extractor can't connect your response to the original message. "Iris said X, Fiet responded Y" becomes split across sessions.

### 3. Single Session Growth

One mega-session eventually needs compaction. CC auto-compacts when context is full. We can capture compact_summary via PostCompact hook, but compaction changes retrieval dynamics.

### 4. Response Routing (SOLVED)

`--output-format json` returns `{ "result": "your reply text", "session_id": "..." }`. Daemon can parse and route to TG/email.

## Four Architecture Proposals

### A. Persistent Stream Daemon
Daemon keeps a persistent `claude -p --input-format stream-json --output-format stream-json` process. Messages fed via stdin JSON, responses read from stdout stream. One long-running CC process, one session.

Pro: Zero startup latency, most natural continuous conversation.
Con: Process management (crash = restart needed), untested stream-json format.

### B. Resume-based Relay (simplest)
Each external message: `claude -p --resume <id> --output-format json "[tg:Iris] msg"`. One session, but each message is a separate process invocation.

Pro: Simplest, most reliable, already tested. Crash-proof (stateless per-call).
Con: 2-3s startup per call. Topic similarity polluted by channel tags. Need queue for concurrency.

### C. Hook-Mediated Inbox/Outbox
TG messages → daemon writes to inbox.jsonl. Daemon sends minimal `[wake:tg]` to CC. UserPromptSubmit hook reads inbox, injects full message as additionalContext. Your response captured by Stop hook's `last_assistant_message`, written to outbox.jsonl. Daemon routes outbox content to TG.

Pro: Session JSONL stays "clean" (user messages are just wake signals). Topic drift not polluted. Fiam can ignore wake-type prompts for similarity.
Con: More complex. inbox/outbox file management. Two hooks needed.

### D. asyncRewake Push
Persistent CC session in tmux (interactive, not -p mode). asyncRewake hook monitors a message queue file. When new messages arrive, hook exits code 2 → CC wakes up with message content in stderr as system reminder.

Pro: Most "alive" feeling — you're always running, messages wake you naturally. Closest to AI autonomy.
Con: Needs persistent tmux, 1.9GB RAM is tight. asyncRewake might not work perfectly for this use case. Interactive mode conflicts with daemon control.

## Key CC Hook Features (from the docs)

- `Stop` hook: receives `last_assistant_message` — your full response text after each turn
- `asyncRewake`: background hook, exit code 2 wakes CC even when idle, stderr shown as system reminder
- `PostCompact`: receives `compact_summary` — session summary after compaction
- `UserPromptSubmit`: `additionalContext` appears in transcript JSONL
- `SessionStart`: can inject additionalContext, set env vars via CLAUDE_ENV_FILE, runs on resume too
