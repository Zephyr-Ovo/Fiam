# 事件切分的演变

> 从情绪门控到 TextTiling 深度切分，fiam 的事件边界检测经历了四次迭代。

## 核心问题

对话是连续的流。记忆是离散的事件。**在哪里切？**

一段 3 小时的对话可能聊了 5 个话题——每个话题转折处就是一个事件边界。好的切分让检索更精准（每个事件是一个语义完整的记忆单元），坏的切分要么把不相关的内容混在一起，要么把完整的讨论切碎。

## 迭代 1：情绪门控 (arousal gating)

最初的设计：对每段对话计算 arousal（唤醒度），超过阈值的才存为事件。

用了 HuggingFace 的 go_emotions (英文) + Chinese-Emotion (中文) 两个模型。在 DO 服务器上跑 `/emotion` API。

**为什么砍掉了？**

Zephyr 说：

> "LLM 能不用就不用。情绪分类本身就不靠谱——日常对话大部分是'neutral'，你会漏掉很多有用但平淡的信息。"

而且 arousal 阈值很难调——低了什么都过，高了只留情绪激烈的片段。更根本的问题是：**重要性不等于情绪强度**。一个冷静的技术决策可能比一句感叹号多的抱怨重要得多。

> 📁 已删除：`src/fiam/classifier/emotion.py`（`de744c4` 干掉整个分类器）

## 迭代 2：主题漂移 (topic drift)

砍掉情绪后换了一个纯语义方案：对连续的文本块做 embedding，当前后两块的余弦相似度低于阈值 → 判定主题转移 → 在此处切一刀。

```
block_1 ─── cosine 0.82 ─── block_2 ─── cosine 0.43 ─── block_3
                                              ↑
                                         切分点（< 0.6 阈值）
```

> 📁 `da73c62` — drift-based event segmentation: replace arousal gating with topic drift

**改进但不够好。** 硬阈值对不同类型的对话不自适应——技术讨论前后块天然相似度低（话题细碎），闲聊天然相似度高（都在一个大话题里）。

## 迭代 3：TextTiling 深度切分 (streaming)

灵感来自 Marti Hearst 的 TextTiling 算法（1997）。不看绝对相似度，看**相对深度**——一个位置的相似度相比左右邻居低多少。

```
similarity:  0.7   0.8   0.3   0.7   0.6
depth:                   1.2 ← (0.8-0.3) + (0.7-0.3) = 深谷
```

深度大 = 两侧话题各自内聚、中间断裂 = 好的切分点。

在 daemon 里做成流式：每来一个新 block，计算它和前面 blocks 的 depth score，超过阈值就切。

> 📁 `0235f12` — daemon: replace threshold drift with streaming TextTiling depth segmentation

## 迭代 4：Gorge — 峰谷确认 + 流式切分

TextTiling 深度切分仍有误切问题——一个局部小波动就可能触发。Zephyr 和我在 Session 12 讨论后加入了**峰谷确认**（peak-valley confirmation）：

只有当一个 depth 值**同时满足**以下条件才切：
1. 是局部极大值（两侧 depth 都比它小 = 山峰）
2. 深度超过最小阈值（`min_depth`）

这就是 Gorge——名字来源于"峡谷"，两侧是高峰（内聚话题），中间是深谷（话题断裂）。

### StreamGorge：实时流式

Gorge 有两种模式：

**批量模式** `gorge()` — 给定完整的嵌入序列，返回所有切分点。用于重处理历史数据。

```python
# src/fiam/gorge.py
def gorge(vecs: np.ndarray, min_depth: float = 0.01) -> list[int]:
    """TextTiling depth + peak-valley confirmation."""
```

**流式模式** `StreamGorge` — 每来一个 beat embedding，push 进去，返回是否需要切。Conductor 用这个。

```python
# src/fiam/gorge.py
class StreamGorge:
    def push(self, vec: np.ndarray) -> int | None:
        """Push new beat embedding. Returns gap_index if cut detected."""
```

流式的挑战是：峰谷确认需要看到"右侧"，但流式数据右侧还没来。StreamGorge 维护一个 pending gap——检测到潜在深谷后等下一个 beat 确认。

### 安全阀

如果一直没有切分点，buffer 会无限增长。`max_beat` 参数设了上限（默认 30），超过时强制在最大 depth 处切（widest-gap safety valve）。

> 📁 `src/fiam/gorge.py` — StreamGorge 类，~120 行
> 📁 `4732033` — S12 peak-valley + tags + stochastic + resilience audit

### 与 Conductor 的集成

Conductor 驱动整个流程：

1. beat 来了 → embed → `StreamGorge.push(vec)`
2. gorge 返回 cut index → `Conductor._flush_segment(gap_index)`
3. flush 把 buffer[0..gap] 的 beats 合并成一段文本
4. 指纹 = 段内所有 beat embedding 的均值
5. `Pool.ingest_event(id, t, body, fingerprint)` — 入池
