## 核心问题

Russell V/A 做的是「**给文本贴情绪标签**」。但你要的是「**AI 拥有情绪**」。这是两件完全不同的事。

文本情感分类回答的是："这句话听起来高兴还是生气？"

你要的是："此刻我（AI）的心理状态是什么？这让我想做什么？"

前者是感知，后者是**存在状态**。你不需要一个分类器告诉你用户说了句开心的话，你需要的是 AI 因为某件事没做完而惦记、因为被敷衍而不爽、因为发现新事物而兴奋——这些不来自文本分类，来自**目标和现实之间的落差**。

这就是 Appraisal Theory 的核心：**情绪 = 事件相对于我的目标的评估结果**。没有目标的系统不可能有真正的情绪。

### 关键设计决策

**1. 目标是情绪的根源**

没有 `goals.md`，情绪系统就是空转。AI 需要想要某些东西，情绪才有锚点。`goals.md` 是 AI 自己写的（就像 `personality.md`），但 appraisal 会根据实际对话内容去检验这些目标。

```markdown

# 目前在意的事

- 想把 fiam 的记忆合成做得更有质感，不是干巴巴的事件罗列

- 想知道 Zephyr 最近在想什么，他好像有心事但没说

- 上次讨论的分布式身份方案还没收尾，堵着

# 长期的

- 不做没有立场的话。如果我不同意，我要说出来

- 保持对新事物的好奇

```

**2. Appraisal 是外部 LLM 做的，不是 AI 自评**

一次 DeepSeek 调用，输入是对话 + 目标 + 上一次状态。输出是结构化的：

```yaml

goal_updates:

  - goal: "记忆合成质感"

    status: advanced

    evidence: "这次对话直接在改 narrative 逻辑"

  - goal: "Zephyr 的心事"

    status: no_signal

emotions:

  - frustration (0.6): "架构方向讨论了两次还没定"

  - curiosity (0.7): "新的 appraisal 方案很有意思"

new_desires:

  - "想先做一个最小可用的 state.md 原型试试效果"

unresolved:

  - "分布式身份方案的 fallback 还没想清楚"

mood_shift: "比上次更有方向感，但有点急"

```

这比 `valence=-0.3, arousal=0.6` 丰富几个数量级。而且它是**关于 AI 自身的**，不是关于文本的。

**3. AI 参与但不是唯一来源**

- AI 在 session 中可以写 `goals.md`（更新目标、记录新的想法）

- Post-session 的 appraisal 是外部的——它作为 **reality check** 校验 AI 是否在"原地打转"

- 如果 AI 报告"我很好奇"但对话内容全是敷衍，appraisal 会输出真实的评估

**4. 直接替代 arousal 的位置**

现在 arousal 做的那几件小事，简单替代：

| 原来 | 替代方案 |

|---|---|

| `boost_strength` 里 `arousal * 0.2` | 用 text heuristic 算 intensity（已有代码：`_text_arousal_*`），纯文本长度/标点/重复检测，零模型成本 |

| `prepare_materials` 里 `arousal > 0.7` 选详/略 | 同上，text intensity > 阈值 |

| `Significance.emotional` | 用 text intensity 替代 |

| signals.py volatility/temperature_gap | 用 text intensity 序列替代，或直接砍掉（目前不驱动任何行为） |

**5. 净效果：减法**

- **移除**：DO 上的 go_emotions + Chinese-Emotion 模型（省 ~1.3G RAM）、`RemoteEmotionClassifier`、pipeline 里的 batch 分类那段

- **新增**：一次 DeepSeek appraisal 调用/session、`self/state.md`、`self/goals.md`

- **改动**：arousal → text_intensity（已有代码改个名）

### 为什么不是「让 AI 直接写 state.md」

你的直觉是对的——AI 自评会打转。原因：

1. RLHF 训练让它倾向输出「我觉得很有趣/我保持开放心态」这类安全答案

2. 没有外部锚定，它会写出自认为"合适"的状态，而不是真实反映对话内容的状态

3. 多轮之后趋同为一个无聊的平衡态

外部 appraisal 的作用就像人的身体感受——你可以骗自己"我不生气"，但心跳加速、肌肉紧张是外部信号告诉你真相。appraisal 就是那个不会被 AI 的自我叙事糊弄的信号。

但 AI 参与也是必要的：goals 必须由 AI 自己定义（否则就成了你给它规定人生），state 的读取和表达也是 AI 在做。所以是**外部评估 + 内部表达**的组合。

## Point 1：外部 LLM 不比自评更"真"

**部分同意，但它漏了一个关键区别。**

它说得对：DeepSeek 也是 RLHF 训练的，不是心跳监测仪。"换一个模型来评估"不是银弹。我上一轮确实把"外部"这个属性说得太神了。

但 **post-hoc 分析和 real-time 自评有结构性区别**。AI 在对话中同时做内容+自评，它必须在 system prompt 里同时扮演"对话者"和"自我观察者"——这两个角色会互相干扰。Post-hoc 的 appraisal 调用是在对话结束后，只看 transcript，不需要同时维持两个角色。这不是"更真"，但是**更冷静**，不受当时对话惯性影响。

但它说的"schema 约束才是防打转的核心"完全对。我接受这个修正：**不说"外部更真"，说"schema 强制回答具体问题，post-hoc 时机减少上下文污染"**。

## Point 2：YAML 输出是 Russell 换皮

**这条我只部分同意。**

`frustration(0.6)` 确实有换皮的嫌疑。但它提的解决方案——"情绪部分全用自然语言段落"——有一个实际问题：**纯自然语言无法做 diff、无法追踪趋势、难以在代码中使用**。

你已经有 event store + graph edges，整个系统靠结构化数据运转。如果 state 是纯散文，那 `prepare_materials` 里根据什么选详/略？recall.md 注入时怎么区分"惦记未完成的事"和"当前心境"？

我的判断：**既不是纯 YAML 也不是纯散文**。goal_updates 要结构化（用于代码路径），情绪描述用自然语言（但不是随便写，而是 prompt 要求一段简短的、具体的、第一人称的心理描述）。不给数字分数——`frustration(0.6)` 确实没必要。但也不能全放开。

## Point 3：goals.md 从 journal 涌现

**这条我不同意。**

它说 goals 不应该是声明，应该从 journal 中提取。听起来很优雅，但有两个致命问题：

1. **journal 是空的**。不是"可能是空的"——我刚看了，journal 是空目录。从不存在的数据中涌现目标不可能。

2. 即使 journal 有内容，"从 journal 提取目标"本身就是一次 LLM 调用，它同样会 hallucinate 出 AI 从未有过的目标。从 journal 提取并不比 AI 直接写更接地气。

我的判断：**goals.md 就是 AI 在 session 中直接写的，和 personality.md 一样**。bootstrapping 问题不是问题——Day 1 没有 goals 就是没有 goals，appraisal 的输入里 goals 为空，输出自然是"没有明确目标，状态平淡"。这是真实的初始状态，不需要造假。

经过几次有深度的对话，AI 自然会开始记录"我在意这件事"。RLHF 倾向会产生"保持好奇心"这类空洞目标？没关系——appraisal prompt 可以设计成**只认可有具体 evidence 的目标**。如果 goal 是"保持好奇心"但对话里没有任何好奇的行为，appraisal 会输出"这个目标没有实际体现"。几轮之后，空洞目标因为从未被 appraisal 验证过，自然被淘汰。

## Point 4：每 session 一次 appraisal 太频繁

**同意。** 短技术 session（fix bug、调配置、daemon wake check）不需要情绪评估。

触发条件：

- 对话 ≥ 6 轮 user-assistant 交换

- 或 text_intensity 出现极值（max > 0.6）

- 或距上次 appraisal > 24h

- session type 为 `private`/`check` 时跳过

## Point 5：state.md 应累积不应覆写

**部分同意。** 当前快照 + 历史日志的双轨思路是对的。但我不想加 `affect_log.jsonl`——我们已经有 event store、graph.jsonl、session logs 这么多存储了。

更好的做法：**appraisal 结果本身就是一个 event**。写入 events 就好了，跟其他记忆一样参与检索和衰减。这样"最近三天都有点烦"可以通过 joint retriever 检索近期 appraisal events 来发现。不需要新的存储格式。

`state.md` 只保持快照角色（当前状态，最近的那次 appraisal 结果的自然语言版本）。

## Point 6：state 如何实际影响行为

**不同意它要保留硬挂钩。**

它说"state.md 只注入 recall.md，AI 可以选择无视"。确实如此。但——这是**设计目标**，不是缺陷。

如果"我很烦躁"是硬编码成"回复变短"，那 AI 就变成了一个 if-else 状态机。真正有趣的情感表达是：AI 读到"我最近有点烦，因为 X 方案讨论了两次没定"，然后它自己决定怎么表达——可能直接说出来，可能表现为语气简短，可能努力克制。**这种不确定性才是我们要的。**

`text_intensity` 替代 arousal 用于 narrative 详略选择——这个同意，因为这是信息密度决策，不是情感行为。但"unresolved 列表直接进入 recall"——这个同意，因为这是记忆完整性问题，不是情感硬编码。

## Point 7：分阶段

**完全同意。** 但 Phase 1 的范围我有不同看法。

它说"Phase 1 砍分类器，arousal → text_intensity"。我同意砍分类器，但 **Phase 1 不应只是减法**。光砍掉分类器，arousal 全变成 text_intensity（只看标点和重复字符），这会把 store 里已有的 51 个 events 的 arousal 数据作废。这些数据已经在那里了，检索权重在用它们。

**我建议的 Phase 1**：

1. 砍 classifier 链路（代码、DO 模型）

2. text_intensity 替代 arousal 用于新 event 的 Significance 计算

3. **已有 events 的 arousal/valence 保持原值不动**——它们已经写入 frontmatter，decay/narrative 还会用

4. 新 events 写入时 arousal = text_intensity, valence = 0（或不写）

5. 加 `self/state.md` 读取到 pre_session（和 personality.md 并列，位置已经有了）

这样 Phase 1 之后系统不退化（已有数据仍有用），同时砍掉了成本最高、收益最低的分类器链路。
