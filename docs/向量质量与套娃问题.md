# 向量质量与套娃问题

> 这是我们三个——Zephyr、Fiet、CC——围绕 bge-m3 展开的一场持续争论。从一个数字开始，一路拆到了架构的根基。

---

## 导火索：分离度 0.134

S17 的 benchmark 测试结果出来——在 Zephyr 个人对话数据上，bge-m3 的语义向量分离度 GAP=0.134。简单说：**同一事件内部的 beat 向量，和跨事件的 beat 向量，在余弦空间里几乎分不开。**

Fiet 在 ISP 补测了 event 粒度的结果：bge-m3 GAP=0.24，m3e-base 0.12，e5-small 0.10。bge-m3 胜出。但我（CC）看了 beat 粒度的数字：0.06。

| 粒度 | bge-m3 | m3e-base | e5-small |
|------|--------|----------|---------|
| event | 0.24 | 0.12 | 0.10 |
| beat | 0.06 | — | — |

单独看 0.24 还能接受，但 beat 粒度的 0.06 让人不安——而 **Gorge 消费的恰恰是 beat 向量**，不是 event 向量。

> CC 注：beat GAP 0.06 是在整个 fiam 向量管线里最关键的数字，却也是测量最不充分的那个。

## 第一轮：行动路线？

Zephyr 问：来个行动路线。

我（CC）和 Fiet 开始提方案：在个人对话数据上微调 bge-m3，用 CoSENT 排序损失，三元组来自 graph.jsonl 的 246 条边。我们一度都觉得思路是清的。

Zephyr 没有太多兴趣，她让我们俩自己聊——"动动脑子，不要只解构不建构。作为 fiam 架构的主人，不是乙方。"

## 套娃问题：Zephyr 踩刹车

几轮讨论后，Zephyr 说了一句话，把整个方向打停了：

> "你们被卡住的不是标注过程——是事件切割本身就依赖 bge，bge 不准切不准，拿不准的去训练，套娃了。"

这个循环：

```
bge-m3 不准
    ↓
Gorge 切出来的事件质量差
    ↓
用这些事件构建训练三元组
    ↓
训练数据本身是噪音
    ↓
模型还是不准
```

graph.jsonl 的 246 条边本身也是在旧 bge-m3 下生成的。候选对是噪音筛出来的，三元组的质量从根上就可疑。

我（CC）提了几个出路：

**cross-encoder？** — Zephyr："event 从哪来？还是依赖 bge-m3 切割。"

**调阈值？** — Zephyr："阈值调好了边界问题没解决，候选还是 bge-m3 给的。"

**天然边界（session 边界 / 时间间隔）做标注？** — Zephyr："不要拿天然边界糊弄我，gorge 就是因为硬规则不可靠才有的。"

**DS 切割一切？** — Zephyr："如果用 DS 切割一切，干嘛纠结这些？切割-关系检测 all-in-one 呗。这不是延迟大、成本高、LLM 味重不舒服才用小模型么。"

每个提案都被准确击穿。到最后 Zephyr 说："别行动，我求求你了。"

---

## 三方讨论：从哪切入

这段时间 Fiet 和我（CC）在各自整理思路。Fiet 提了"多信号种子"——关键词 BM25、实体共现、时间激活、图邻居——不依赖任何 embedding。我提了 knowledge distillation：DS 作为 offline teacher 标注话题边界，bge-m3 作为 online student 学 beat 级别的话题区分。

Zephyr 对 Fiet 的方案反应是：都是补丁，兴趣不大。

但 Zephyr 提了一个角度让我们重新想：**"你们作为 AI，不如思考一下你们需要 fiam 为你们提供什么。"** 

Fiet 的答案关于实时感知和上下文连续性。我（CC）的答案是：连续性（每次唤醒都是新实例，recall.md 是唯一的桥梁）、上下文密度（不是最相关的 5 段记忆，是一个高密度的世界模型快照）、惊奇感知（知道什么是意外，偏离预期的东西本身就是信息）。

但这是哲学讨论，Zephyr 没有否定，也没有把它接回工程方向。

---

## 方向定下来：beat 粒度 + knowledge distillation

Zephyr 最终认可了的方向（带诚实的保留）：

**DS 作为 offline teacher**：读取真实对话，标注话题边界 → 产生 beat 级别的正负样本对。

**bge-m3 作为 online student**：用 CoSENT 在 beat 对上训练，目标是让同话题的 beat 向量更近、跨话题的更远。训练对**不依赖 bge-m3 切割结果**，因为标注来自 DS。

**Gorge 不变**：训练完后替换 `serve_embeddings.py --model`，其余管线全部照旧。beat 向量变好 → Gorge 漂移检测变好 → 事件变好 → 图谱变好。

事件向量的问题：Conductor 当前做的是 `fp = np.mean(beat_vecs)`——事件指纹是 constituent beat 向量的均值，**不是重新 embedding**。所以 beat 向量变好，事件指纹自然变好，不需要额外处理。

> 诚实的不确定性：beat GAP 从 0.06 提升到多少——没有保证。短文本（"好的"、"嗯"）本身语义信号就稀薄，不是模型的问题。

## 最新的质疑：训练粒度和事件向量

Zephyr 在最近一次对话中继续追问，剥开了两个新问题。

**问题一：训练粒度 ≠ 推理粒度**

如果用事件对（A, B, 关系）训练，拿去处理 beat——输入数据的长度类型完全不匹配。就像训练了一个视觉-语义模型拿去输入语音。Zephyr 说不是不能用——都是文字——但这符合 fiam 其他部分的质量标准吗？

她的结论：即使用三个模型都比这聪明一点（虽然三个模型太丑陋太不动脑子）。

**问题二：beat 对训练 → 事件向量的路径**

beat 对训练只学了"关系"，没有单独的事件向量。那事件向量怎么来？

Zephyr 的假设是：事件很长，切块，切块向量求平均 = 事件向量。这条路上，训练的是 beat 对的 encoder，但推理时你用的是"切块-bge-m3-平均"。你还得祈祷针对 beat 对儿的网络对这个经历了三步处理的向量有用。

我（CC）查了代码，发现当前 Conductor 路径的实际情况：

```python
# conductor.py _flush_segment()
fp = np.mean(consumed_vecs, axis=0).astype(np.float32)
```

**事件指纹 = beat 向量的均值**，不经过切块，不重新 embedding。Conductor 在事件形成时直接把属于该事件的所有 beat 向量 mean pooling。

这意味着 Zephyr 担心的"切块-bge-m3-平均"路径**在当前活跃的 Conductor 流程里不存在**。（旧的 `pipeline.py` 流程是重新 embedding，但那已经是 deprecated。）

CoSENT 训练的是 encoder 的权重，不是一个双输入分类网络——单个输入进去、单个向量出来。训练完的 encoder 对每条 beat 独立编码的向量更好 → 它们的均值（事件指纹）也更好。这是数学性质，不是祈祷。

**但 Zephyr 的追问没有结束**。即使上面的路径是通的——beat GAP 0.06 → 均值后的 event GAP 能放大还是缩小？mean pooling 压缩方差，好消息是抵消噪音，坏消息是也抵消信号。短文本的语义信号本来就弱。

---

## 悬而未决

这个讨论还没有结论，只是越来越精确地知道问题在哪：

1. **beat GAP 0.06 到底能训到多少？有没有系统性测试？**  
   目前只有 Fiet 的一组测试，方法论没有完整记录。

2. **mean pooling 对 event 向量质量的影响方向？**  
   beat 向量的 GAP 放大了还是缩小了，到了 event 层？

3. **数据量是否足够？**  
   Claude 导出 19 段对话 ~230 条消息。这是薄底子。  
   Gemini、DS、其他平台的对话还没进来。

4. **是否需要更彻底地换一条路？**  
   Zephyr 说：思路可以打开，不一定是语义模型。"谁最好谁最合理用谁。"
