# 记忆图谱与检索

> 事件之间的关系网络，以及如何通过扩散激活找回记忆。

## 为什么要图？

向量检索（top-k cosine）只能找到语义相似的内容。但人的记忆不只是"相似联想"——昨天吃的饭可能让你想起三年前在同一家餐厅的求婚，不是因为"吃饭"和"求婚"语义相近，而是因为**地点关联**。

fiam 的记忆图谱让事件之间建立多种类型的关系边，检索时沿着这些边传播激活，就像神经元的兴奋扩散。

## 边的类型

| 类型 | 含义 | 权重来源 |
|------|------|----------|
| `temporal` | 时间相邻 | 指数衰减：$w = e^{-\Delta t / \tau}$ |
| `semantic` | 语义相似 | cosine similarity |
| `cause` | 因果关系 | DeepSeek 判断 |
| `remind` | 联想触发 | DeepSeek 判断 |
| `elaboration` | 展开/细化 | DeepSeek 判断 |
| `contrast` | 对比/反差 | DeepSeek 判断 |

前两种是算法自动生成的，后四种需要语义理解，交给 DeepSeek 判断。

> Zephyr 的原则："LLM 在边类型判断上可以用，因为这不在关键路径上，且确实需要语义理解。但切分、漂移、检索的核心逻辑必须是算法。"

## 边的生成流程

新事件入池时，`graph_builder.build_edges()` 执行：

1. **Temporal 边** — 与时间上最近的 N 个事件建边，权重按时间差指数衰减
2. **Semantic 候选** — 查 cosine.npy 矩阵，取相似度 top-k 候选
3. **DS 判断** — 对候选对调用 DeepSeek：判断边类型 + 权重 + 确认/否决
4. **写入** — 更新 PyG 格式的 `edge_index.npy` + `edge_attr.npy`

同时 DS 会给新事件命名（英文 snake_case），作为 event ID。

> 📁 `src/fiam/retriever/graph_builder.py` — `build_edges()` 函数
> 📁 Conductor 在 `_post_ingest()` 中自动调用

## 检索：扩散激活

不是 top-k。不是 MMR。是**图扩散激活**（spreading activation）。

### 核心算法

```
1. Seed   — 用当前滑动向量在 fingerprints.npy 找最近的事件
           （今日事件屏蔽——还在 session 内，不需要"回忆"）

2. Spread — 从 seed 沿所有出边传播
           激活值 = 父节点激活 × 边权重
           低于阈值 (0.4) 停止
           BFS 展开，每个节点最多被一条路径激活

3. Select — 每个被激活的节点以激活值为概率独立决定是否 recall
           NOT top-k: 可能返回 0 个、也可能返回 5 个
           概率选择意味着同一个查询两次的结果不完全一样——这很像人
```

> 📁 `src/fiam/retriever/spread.py` — `retrieve()` 函数

### 为什么不用 top-k？

Top-k 有几个问题：
1. 永远返回 k 个结果——即使没有真正相关的记忆
2. 排序是确定性的——同一个查询永远返回同一组结果，不自然
3. 只看语义距离——忽略了图结构中的关联路径

扩散激活的优势：
- 沿边传播意味着**间接关联**也能被触及（A→B→C，A 和 C 语义不近但有路径）
- 概率选择引入了自然的**随机性**——就像人有时候突然想起一件不太相关的事
- 激活阈值自动过滤弱关联——不会硬凑 k 个结果

### 权重连乘的衰减

传播路径越长，激活值越低（权重连乘）。这自然实现了"远距离关联弱于近距离关联"的直觉。

```
seed (1.0) → A (0.8) → B (0.8×0.6=0.48) → C (0.48×0.3=0.14 < 0.4, 停)
```

### 今日事件屏蔽

`shield_after` 参数：今天凌晨之后的事件不作为 seed 候选。因为这些事件还在当前 session 的 context 里，不需要"回忆"。

## 历史演变

### Phase 1: 4-factor scoring (`joint.py`)

最早的检索是四因素加权：
- 语义相似度 (embedding cosine)
- 记忆强度 (Ebbinghaus retention)
- 图中心度 (PageRank-like)
- MMR 多样性惩罚

问题：权重怎么调？四个因素的平衡全靠经验。

### Phase 2: min_score gate

加了 min_score 门控（0.15），语义相似度太低的直接过滤。减少了噪声但没解决根本问题。

### Phase 3: 图扩散激活

在 S14 中彻底重写。`joint.py`、`graph.py`(NetworkX)、`decay.py`(Ebbinghaus) 全部被 `spread.py` 一个文件替代。

## Hebbian 强化（待实现）

计划中的特性：当两个事件在同一次 recall 中被共同激活，增强它们之间边的权重。类似神经科学中的 "cells that fire together wire together"。

这会让记忆图谱随着使用逐渐自我优化——经常一起被想起的事件之间的连接会越来越强。
